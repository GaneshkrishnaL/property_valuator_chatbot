RAG Property Valuation Chatbot

Complete Architecture & Code Guide for your Google Cloud CE Interview

The Big Picture - What This Project Actually Does

Imagine a large real estate company that has thousands of PDF documents: appraisal reports, property listings, market analyses, comparable sales reports. All their valuable data is trapped inside these PDFs in messy formats that tools like ChatGPT or LangChain can't properly understand.

This project builds a custom RAG (Retrieval-Augmented Generation) pipeline that solves this problem in two main phases:

Think of it like a library system:

Phase 1 - INGESTION (run once): A librarian reads every PDF, understands what type of document it is, extracts the important information, organizes it into neat index cards, and files them in a searchable cabinet.

Phase 2 - QUERY (run every time a user asks): When someone asks "What's the value of 742 Evergreen Terrace?", the librarian searches the cabinet, pulls out the most relevant cards, and writes a well-sourced answer using only what's on those cards (no making stuff up).

Config

Central Configuration

config.py
WHAT IT DOES

This is the brain's settings panel. Every tunable value in the entire project lives here. Instead of hardcoding numbers like "search for top 10 results" scattered across 15 files, they're all centralized so you can tweak behavior from one place.

KEY SETTINGS

Backend flags: USE_VERTEX_AI and USE_GEMINI control whether you use local models (for demo) or Google Cloud (for production). Flip these to True and the entire system switches to GCP services.

Retrieval tuning: CHUNK_SIZE=800 (how big each text chunk is), CHUNK_OVERLAP=100 (overlap between chunks so you don't cut sentences), TOP_K=10 (how many results to fetch), SIMILARITY_THRESHOLD=0.3 (minimum relevance score).

ChromaDB path: Configurable via environment variable so it works on different machines without disk errors.

# The two flags that switch between demo and production
USE_VERTEX_AI = False   # True = use Google Vertex AI embeddings
USE_GEMINI = False      # True = use Gemini Pro instead of OpenAI

# Retrieval tuning - these control search quality
TOP_K = 10              # Fetch top 10 most relevant chunks
SIMILARITY_THRESHOLD = 0.3  # Ignore results below 30% relevance
Interview tip: When the interviewer asks "How would you move this to production?", you say "I flip USE_VERTEX_AI and USE_GEMINI to True in config.py, set the GCP project ID, and the entire system switches from local models to Vertex AI embeddings and Gemini Pro. Zero code changes needed."

Pipeline

PDF Extractor

pipeline/pdf_extractor.py
WHAT IT DOES

This is Step 1 of the pipeline. It opens each PDF file and pulls out two separate things: the raw text AND any tables it finds. Most tools just give you flat text. This one preserves the difference between "a paragraph of description" and "a table of comparable sales data".

HOW IT WORKS - STEP BY STEP

Step 1: Opens the PDF using PyMuPDF (imported as fitz). This library can read any PDF, even ones with complex layouts.

Step 2: For each page, extracts the raw text using page.get_text().

Step 3: Filters out headers and footers. Lines containing "CONFIDENTIAL", "Page X |", or "Generated on" are removed so they don't pollute your embeddings later. Without this, every single chunk would have "CONFIDENTIAL" in it, confusing the search.

Step 4: Detects and extracts tables using PyMuPDF's built-in table finder. Each table becomes a list of rows, where each row is a list of cell values. So a 3-column, 5-row comparable sales table becomes a Python list of 5 lists, each with 3 values.

Step 5: Returns an ExtractedDocument object containing pages (each with text + tables), the full concatenated text, and metadata like filename and page count.

# What the output looks like conceptually:
ExtractedDocument(
    pages=[
        ExtractedPage(
            text="Property at 742 Evergreen Terrace is a 4-bed...",
            tables=[
                [["Address", "Price", "Sqft"],       # header row
                 ["750 Elm St", "$535,000", "2,380"],  # data rows
                 ["801 Oak Ave", "$612,000", "3,100"]]
            ]
        )
    ],
    full_text="Property at 742 Evergreen...",
    metadata={"filename": "appraisal_742_evergreen.pdf", "pages": 3}
)
Production upgrade: The file contains commented-out code showing how to swap PyMuPDF for Google Document AI. Document AI adds OCR for scanned documents, ML-based table extraction, and entity detection.
Pipeline

Document Classifier

pipeline/document_classifier.py
WHAT IT DOES

After extraction, we need to figure out WHAT TYPE of document we're looking at, because each type needs a different parser. An appraisal has "assessed value" and "comparable sales table" while a listing has "MLS#" and "days on market".

HOW IT WORKS

Think of it like a scoring game. For each document type, there's a set of "signal" patterns, each with a weight:

For APPRAISAL: if the text contains "appraisal report" (worth 0.35 points), "appraiser" (0.20), "assessed value" (0.15), etc.

For LISTING: "property listing" (0.30), "MLS#" (0.25), "list price" (0.20), etc.

For MARKET_ANALYSIS: "market analysis" (0.35), "year-over-year" (0.15), etc.

For COMPARABLE_SALES: "comparable sales report" (0.35), "$/sq ft" (0.15), etc.

It also adds structural bonuses. A document with 2+ pages is more likely an appraisal. Lots of tables? Probably comparable sales. The type with the highest total score wins.

# Example: classifying a document
result = classify_document(extracted_text)

# result might be:
ClassificationResult(
    document_type=DocumentType.APPRAISAL,
    confidence=0.87,
    reasoning="Strong appraisal signals: 'appraisal report' (0.35), 'appraiser' (0.20)..."
)
Why this matters: Classification determines which PARSER handles the document next. Without it, you'd need one giant parser that handles every document type. That would be fragile and miss type-specific fields.
Pipeline

Custom Parsers - THE CORE OF THE PROJECT

pipeline/custom_parsers.py
WHAT IT DOES

This is the most important file in the entire project. It's the reason the prototype exists. It solves the core problem: "proprietary data trapped in unstructured PDFs."

There are 4 parser classes, one for each document type. Each parser knows exactly what fields to look for and how to extract them from that specific document format.

THE 4 PARSERS

AppraisalParser extracts: property_address, square_footage, bedrooms, bathrooms, lot_size, year_built, assessed_value, estimated_value, comparable sales from tables, and splits text into sections (Introduction, Assessment, Conclusion).

ListingParser extracts: listing_price, MLS number, days_on_market, agent info, property features. It knows that a listing's price comes after "List Price:" while an appraisal's value comes after "Estimated Market Value:".

MarketAnalysisParser extracts: neighborhood statistics, trends, median prices, year-over-year changes. It parses the statistical tables and creates per-neighborhood summaries.

ComparableSalesParser extracts: individual sale records from the big table, with address, price, date, sqft, and price-per-sqft for each sale.

# Each parser outputs structured fields + text sections
result = appraisal_parser.parse(extracted_doc)

# result looks like:
{
    "structured_fields": {
        "property_address": "742 Evergreen Terrace, Austin, TX",
        "bedrooms": 4,
        "estimated_value": "$575,000",
        "comparable_sales": [...]
    },
    "text_sections": [
        {"title": "Property Description", "content": "The property is..."},
        {"title": "Valuation Conclusion", "content": "Based on our analysis..."}
    ]
}
Interview tip: This is your strongest talking point. "Standard RAG tools like LangChain treat PDFs as flat text. Our custom parsers understand the document structure, so when someone asks about 3-bedroom homes, we can filter by the actual bedroom count in metadata, not just hope the word '3-bedroom' appears near the right content."
Pipeline

Smart Chunker

pipeline/chunker.py
WHAT IT DOES

After parsing, we have structured data and text sections. But you can't embed an entire 3-page document as one vector. It would be too general and wouldn't match specific queries well. So the chunker splits each document into smaller, focused pieces while keeping important metadata attached.

HOW IT WORKS

Semantic chunking (not dumb splitting): Instead of cutting every 800 characters blindly, it splits along section boundaries. The "Property Description" section becomes one chunk, the "Valuation Conclusion" becomes another. This means each chunk is about one topic.

Paragraph-boundary splitting: If a section is longer than 800 chars, it splits at paragraph boundaries (double newlines) rather than mid-sentence. The overlap of 100 chars ensures no information is lost at boundaries.

Metadata attachment: Every chunk gets rich metadata from the parser: property_address, neighborhood, bedrooms, bathrooms, square_footage, price, year_built, and document_type. This is crucial for hybrid search.

Structured summary chunk: For property documents, it creates an extra "summary" chunk that lists all the structured fields in a readable format. This chunk is great for direct-match queries like "tell me about 742 Evergreen".

# A single chunk looks like this (Pydantic model):
DocumentChunk(
    text="The subject property at 742 Evergreen Terrace features...",
    metadata={
        "source_file": "appraisal_742_evergreen.pdf",
        "document_type": "appraisal",
        "property_address": "742 Evergreen Terrace, Austin, TX",
        "neighborhood": "South Congress",
        "bedrooms": 4,
        "bathrooms": 3,
        "price": 575000,
        "section": "Property Description"
    },
    chunk_id="appraisal_742_evergreen_chunk_0"
)
Key insight: The metadata is what makes hybrid search possible. When someone asks "3-bedroom homes under $600K in Downtown Austin", we can FILTER by bedrooms=3, price<600000, neighborhood=Downtown Austin BEFORE doing vector similarity search. This is much more accurate than relying on embeddings alone.
Retrieval

Embedding Model

retrieval/embeddings.py
WHAT IT DOES

Converts text chunks into number vectors (arrays of 384 numbers) so you can measure how "similar" two pieces of text are mathematically. The sentence "What's this house worth?" and the chunk about "Valuation Conclusion: estimated at $575,000" end up as vectors that are close together in 384-dimensional space.

HOW IT WORKS

Local mode (demo): Uses sentence-transformers/all-MiniLM-L6-v2, a small but effective model that runs entirely on your laptop. It produces 384-dimensional vectors. No API key needed, works offline.

Production mode: Uses Google Vertex AI's text-embedding-004 model, which produces 768-dimensional vectors. Better quality, but requires GCP credentials and API calls.

The code uses a Protocol (Python's version of an interface) so both implementations have the same embed() and embed_batch() methods. A factory function get_embedding_model() checks the config flag and returns the right one.

# The model converts text to a list of numbers
model = get_embedding_model()  # returns LocalEmbeddingModel or VertexAI

vector = model.embed("What is this property worth?")
# vector = [0.023, -0.118, 0.445, ..., 0.067]  (384 numbers)

# Similar questions produce similar vectors
# "property valuation" and "how much is it worth" → vectors close together
# "property valuation" and "best pizza in Austin" → vectors far apart
Retrieval

Vector Store (ChromaDB)

retrieval/vector_store.py
WHAT IT DOES

This is the "searchable filing cabinet" from our library analogy. It stores all the embedded chunks and lets you search them by vector similarity (which chunks are most relevant to a query) and by metadata filters (only show me 3-bedroom homes).

HOW IT WORKS

ChromaDB is a lightweight vector database that runs locally. No server needed, it just saves files to disk. It has 3 main operations:

add_chunks() - Takes document chunks, converts them to vectors using the embedding model, and stores them with their metadata. Each chunk gets a unique ID.

search() - Takes a query vector and optional metadata filters (a "where" clause). Returns the top-K most similar chunks. It uses cosine similarity: chunks whose vectors point in the same direction as the query vector score highest.

reset() - Deletes everything so you can re-ingest from scratch.

# Store chunks during ingestion
vector_store.add_chunks(all_chunks, embedding_model)

# Search with a metadata filter
results = vector_store.search(
    query_vector=[0.023, -0.118, ...],  # 384 numbers
    where={"bedrooms": 3, "neighborhood": "Downtown Austin"},
    n_results=10
)
# Returns: top 10 chunks about 3-bed Downtown Austin properties
Production upgrade: In production, you'd swap ChromaDB for Vertex AI Vector Search. It handles millions of vectors, auto-scales, and integrates with GCP's security and monitoring. The swap requires minimal code changes because of the abstraction layer.
Retrieval

Hybrid Search Engine

retrieval/search.py
WHAT IT DOES

This is the smartest part of the retrieval system. Instead of just doing vector similarity search (which is what most RAG demos do), it combines three strategies: metadata filtering, vector similarity, and result reranking.

THE 3-STAGE SEARCH PROCESS

Stage 1 - Extract Filters from Query: The function extract_query_filters() parses the natural language query using regex patterns. "3-bedroom homes under $600K in Downtown Austin" becomes QueryFilters(bedrooms=3, max_price=600000, neighborhood="Downtown Austin"). These filters narrow down the search space before any vector comparison.

Stage 2 - Vector Search with Filters: The extracted filters become a ChromaDB "where" clause. ChromaDB first filters chunks by metadata (only 3-bed, under $600K, Downtown Austin), then ranks the remaining chunks by vector similarity to the query. This is much more accurate than searching all chunks.

Stage 3 - Reranking: The raw results get bonus scores for specific matches. If a chunk's address matches the query (+0.15 boost), if the neighborhood matches (+0.08), if the document type matches the query intent (+0.05). Results are re-sorted by the boosted scores.

# What happens when you search for "3-bedroom homes in Downtown Austin under 600K"

# Stage 1: Extract filters
filters = extract_query_filters(query)
# QueryFilters(bedrooms=3, max_price=600000, neighborhood="Downtown Austin")

# Stage 2: Build ChromaDB where clause
where = build_chroma_where(filters)
# {"$and": [{"bedrooms": 3}, {"price": {"$lte": 600000}}, ...]}

# Stage 3: Search + Rerank
results = hybrid_search(query, embedding_model, vector_store)
# Returns SearchResult objects sorted by boosted relevance score
Interview tip: "Pure vector search struggles with numerical queries. If someone asks for homes under $600K, vector similarity might return a $1.2M mansion because the text descriptions are semantically similar. Our hybrid approach uses metadata filters first to ensure numerical constraints are respected, then vector similarity for semantic matching."
Chatbot

LLM Client

chatbot/llm_client.py
WHAT IT DOES

This file handles communication with the large language model (LLM). It sends the assembled prompt (system instructions + retrieved documents + user question) and gets back a natural language answer.

HOW IT WORKS

Demo mode: Uses OpenAIClient which calls GPT-4o-mini via the OpenAI API. Temperature is set to 0.3 (low = more factual, less creative). Has both generate() (full response at once) and generate_stream() (token-by-token for typing effect).

Production mode: Uses GeminiClient which calls Gemini Pro via Vertex AI. Same interface, different backend. The factory function get_llm_client() reads the config flag and returns the right client.

Both clients implement the same protocol (interface), so the rest of the code doesn't care which one is active. This is the Strategy pattern from software design.

client = get_llm_client()

# Same interface regardless of backend
response = client.generate(
    system_prompt="You are a property valuation expert...",
    user_prompt="Based on these documents: [chunks]... What is 742 Evergreen worth?"
)
# response = "Based on the appraisal report, the estimated market value..."
Chatbot

Prompt Templates

chatbot/prompt_templates.py
WHAT IT DOES

Different types of questions need different instructions to the LLM. A valuation question ("What's this house worth?") needs different guidance than a market trends question ("Are prices going up?"). This file contains query-specific prompt templates that guide the LLM to give the right kind of answer.

THE 4 TEMPLATES

VALUATION_TEMPLATE: Tells the LLM to look for stated values first ("the appraisal says $575,000") before computing anything. Prevents the LLM from inventing estimates using price-per-sqft math.

COMPARISON_TEMPLATE: Instructs structured side-by-side comparison of properties (price, size, condition, neighborhood).

MARKET_ANALYSIS_TEMPLATE: Focuses on trends, statistics, and neighborhood-level insights.

GENERAL_TEMPLATE: Catch-all for other questions. Tells the LLM to present even single results in detail rather than saying "insufficient data".

HALLUCINATION PREVENTION

The system prompt contains strict rules: "Only use information from the provided context", "If you don't have enough information, say INSUFFICIENT DATA", "Always cite your sources with [Source: filename]". These instructions prevent the LLM from making up property values or inventing market trends.

The function classify_query() reads the user's question and picks the right template based on keywords. "value", "worth", "price estimate" triggers VALUATION. "compare", "versus", "difference" triggers COMPARISON. And so on.

# Query classification example
template = classify_query("What is 742 Evergreen Terrace worth?")
# Returns VALUATION_TEMPLATE because "worth" is a valuation keyword

template = classify_query("Compare homes in South Congress vs Downtown")
# Returns COMPARISON_TEMPLATE because "compare" is a comparison keyword
Chatbot

Main Orchestrator

chatbot/chatbot.py
WHAT IT DOES

This is the conductor of the orchestra. When a user asks a question, this file coordinates the entire query flow: search for relevant documents, pick the right prompt template, call the LLM, verify the answer's numbers, extract source citations, and assess confidence. Everything flows through the PropertyValuationChatbot.ask() method.

THE ask() METHOD - STEP BY STEP

Step 1 - Search: Calls hybrid_search() with the RAW query (not enriched with history). This was a critical bug fix. Originally, conversation history was appended to the query before searching, which meant filters from previous questions contaminated the current search.

Step 2 - Classify: Determines what type of question this is (valuation, comparison, market, general) and picks the matching prompt template.

Step 3 - Context check: If no relevant documents were found above the similarity threshold, returns "INSUFFICIENT DATA" immediately without calling the LLM. This saves API costs and prevents hallucination on empty results.

Step 4 - Build prompt + add history: Assembles the system prompt + template + retrieved chunks + user question. Conversation history is ONLY added here (to the LLM prompt), not to the search. This way the LLM understands follow-up questions but the search isn't contaminated.

Step 5 - Call LLM: Sends the assembled prompt to GPT-4o-mini (or Gemini) and gets the response.

Step 6 - Verify numbers: Extracts all dollar amounts from the LLM's response and checks each one against dollar amounts found in the source documents. If a number is more than 20% away from any source number, it's flagged as "unverified". This catches hallucinated values.

Step 7 - Extract sources and confidence: Pulls "[Source: filename]" citations from the response. Assesses confidence based on result count, similarity scores, and whether any numbers were unverified.

# The critical fix: raw query for search, history only for LLM

# Step 1: Search with RAW query (no history contamination)
results = hybrid_search(
    query=query,  # "Compare homes in South Congress" - just this
    embedding_model=self._embedding_model,
    vector_store=self._vector_store,
)

# Step 4: History is ONLY added to the LLM prompt
system_prompt, user_prompt = build_prompt(query, results)
if self._history.messages:
    context_hint = self._history.get_context_hint()
    user_prompt = user_prompt + "\n" + context_hint
# Now the LLM knows about prior Q&A but search wasn't affected
Why the history bug was critical: Without this fix, if Query 2 asked about "3-bedroom homes in Downtown Austin under $600K", then Query 3 about "Compare 742 Evergreen Terrace" would ALSO filter for 3 bedrooms, Downtown Austin, and under $600K, because the history text contained those words. The search would find nothing because 742 Evergreen is in South Congress with a $575K price. Isolating search from history fixed this.

End-to-End Example: What Happens When You Ask a Question

Let's trace what happens when you type: "What is the estimated value of 742 Evergreen Terrace?"

Step 1 - chatbot.py receives the query
The ask() method gets called with your raw question string.
Step 2 - search.py extracts filters
extract_query_filters() scans the query with regex. It finds "742 Evergreen Terrace" matches an address pattern but no bedroom/price filters. Returns QueryFilters(address="742 Evergreen Terrace").
Step 3 - search.py runs hybrid search
The query text gets embedded into a 384-number vector. ChromaDB searches all stored chunks, filtering by address metadata where possible, and returns the top 10 most similar chunks. The appraisal chunks for 742 Evergreen score highest because they're about that specific property.
Step 4 - search.py reranks results
Chunks that mention "742 Evergreen" in their address metadata get a +0.15 score boost. Appraisal documents get a bonus because the query is about valuation. Results are re-sorted.
Step 5 - prompt_templates.py classifies the query
The word "value" triggers the VALUATION template. This template tells the LLM: "Look for stated values in the documents first. Don't compute your own estimate."
Step 6 - chatbot.py builds the prompt
Assembles: system prompt (you are a property valuation expert) + valuation template + the top retrieved chunks as context + the user's question. If there's conversation history, it's appended as a hint.
Step 7 - llm_client.py calls GPT-4o-mini
The full prompt is sent to OpenAI's API. GPT-4o-mini reads the retrieved chunks (which contain the appraisal's "Valuation Conclusion: estimated market value of $575,000") and generates a natural language answer citing that value.
Step 8 - chatbot.py verifies numbers
The response mentions "$575,000". The verification function checks: does $575,000 appear in the source chunks? Yes, it's in the appraisal conclusion. Verified. If the LLM had said "$999,000" (not in any source), it would be flagged as unverified.
Step 9 - cli_demo.py displays the answer
The Rich library renders a color-coded panel with the answer, a "HIGH" confidence badge (green), the source citation "[Source: appraisal_742_evergreen.pdf]", and the extracted filters used for search.
