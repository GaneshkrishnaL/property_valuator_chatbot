Create Your Google Cloud Account

Step 1.1: Sign up for Google Cloud

Go to https://cloud.google.com and click "Start free". Sign in with your Google account. You'll get $300 in free credits valid for 90 days. You need a credit card for verification, but you WON'T be charged unless you manually upgrade to a paid account.

What is Google Cloud? It's Google's collection of cloud services. Think of it like renting Google's computers, databases, and AI models instead of running everything on your laptop. You pay only for what you use, and most services have a free tier.

Step 1.2: Create a Project

A "project" in GCP is like a folder that holds all your services, billing, and permissions together. Everything we build goes into one project.

In the GCP Console (the web dashboard), click the project dropdown at the top and click "NEW PROJECT":

Project name: property-valuation-chatbot
Organization: (leave as "No organization" if you don't have one)
Why a project? GCP organizes everything by project. Billing is tracked per-project, permissions are set per-project, and when you're done experimenting, you can delete the entire project and everything gets cleaned up. Very neat.

Step 1.3: Install the Google Cloud CLI (gcloud)

This is the command-line tool for talking to Google Cloud from your Mac's terminal.

# On Mac, install using Homebrew (easiest way)
brew install google-cloud-sdk

# If you don't have Homebrew, install it first:
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# After installing gcloud, log in to your Google account:
gcloud auth login
# This opens a browser window. Sign in with your Google account.

# Set your project as the default:
gcloud config set project property-valuation-chatbot
What is gcloud? It's like a remote control for Google Cloud. Instead of clicking through the web dashboard, you type commands. gcloud config set project ... tells gcloud "every command I run from now on should target this project."

Step 1.4: Enable the APIs we need

GCP services are "off" by default. You need to enable each service you want to use. Think of it like turning on switches.

# Enable all the services we'll use (run this as one command)
gcloud services enable \
  aiplatform.googleapis.com \
  documentai.googleapis.com \
  run.googleapis.com \
  cloudbuild.googleapis.com \
  storage.googleapis.com \
  secretmanager.googleapis.com \
  firestore.googleapis.com \
  artifactregistry.googleapis.com
What are APIs? Each GCP service (Vertex AI, Document AI, Cloud Run, etc.) exposes an API (Application Programming Interface). Enabling the API means "I want to use this service in my project." It doesn't cost anything to enable them. You only pay when you actually USE them.

Here's what each one does:

aiplatformVertex AI - embeddings and Gemini LLM
documentaiDocument AI - PDF OCR and table extraction
runCloud Run - hosts your app as a web service
cloudbuildCloud Build - builds your Docker container in the cloud
storageCloud Storage - stores your PDF files
secretmanagerSecret Manager - stores API keys securely
firestoreFirestore - database with vector search
artifactregistryArtifact Registry - stores your Docker images

Set Up Cloud Storage for PDFs

Step 2.1: Create a Storage Bucket

A "bucket" is like a folder in the cloud that stores files. We'll put our PDFs here.

# Create a bucket (name must be globally unique, so add your initials)
gcloud storage buckets create gs://pvr-chatbot-ganesh \
  --location=us-central1 \
  --uniform-bucket-level-access

# Create subfolders for organizing
gcloud storage cp /dev/null gs://pvr-chatbot-ganesh/raw-pdfs/
gcloud storage cp /dev/null gs://pvr-chatbot-ganesh/processed/
What is Cloud Storage? Think of it like Google Drive, but for your application. Your app can upload, download, and list files programmatically. The --location=us-central1 means your files are stored in Google's data center in Iowa, USA. You can change this to asia-south1 (Mumbai) if you want data closer to India.

Step 2.2: Upload your PDFs

# Upload the generated sample PDFs

# Later, you can upload your real Indian property docs too:

Set Up Vertex AI (Embeddings + Gemini)

Step 4.1: Enable Vertex AI and test Gemini

# Vertex AI was already enabled in Step 1.4
# Let's set the default region
gcloud config set ai/region us-central1

# Test that Gemini works from the command line:
gcloud ai models describe gemini-2.0-flash --region=us-central1
What is Vertex AI? It's Google's unified AI platform. It hosts embedding models (converting text to vectors), LLMs like Gemini (generating answers), and vector search (finding similar vectors). In our chatbot, it replaces both the local sentence-transformers model AND OpenAI GPT-4o-mini. Everything stays within Google's ecosystem.

Step 4.2: Set up authentication for your app

# Create a "service account" - this is like a username for your app
gcloud iam service-accounts create pvr-chatbot-sa \
  --display-name="Property Valuation Chatbot"

# Give it permission to use the services it needs
PROJECT_ID=$(gcloud config get-value project)

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:pvr-chatbot-sa@$PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:pvr-chatbot-sa@$PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/documentai.editor"

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:pvr-chatbot-sa@$PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/storage.objectViewer"

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:pvr-chatbot-sa@$PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/datastore.user"

# Download a key file for local testing
gcloud iam service-accounts keys create key.json \
  --iam-account=pvr-chatbot-sa@$PROJECT_ID.iam.gserviceaccount.com

# Set the environment variable so your Python code can authenticate
export GOOGLE_APPLICATION_CREDENTIALS="key.json"
What is a Service Account? When YOU use GCP, you log in with your Google account. But when your APP uses GCP, it needs its own identity. A service account is like a "robot user" for your app. You give it specific permissions (can use Vertex AI, can read Cloud Storage, etc.) and it gets a key file to prove its identity. This is how your Python code authenticates with Google Cloud services.
Security warning: Never commit key.json to git! Add it to your .gitignore. When deployed on Cloud Run, the service account authenticates automatically without a key file.

Update Your Code for GCP

Step 5.1: Update config.py

Flip the flags we built into the architecture from day one:

# config.py - change these two lines:
USE_VERTEX_AI = True    # was False
USE_GEMINI = True       # was False

# Add your GCP project details:
GCP_PROJECT_ID = "property-valuation-chatbot"
GCP_LOCATION = "us-central1"
DOCUMENT_AI_PROCESSOR_ID = "your-processor-id-from-step-3"
Remember this from the architecture guide? The entire codebase was designed with this moment in mind. The embedding model, LLM client, and PDF extractor all have factory functions that check these flags and return the appropriate implementation. Two lines change, the whole system switches to GCP.

Step 5.2: Update requirements.txt

# Add these GCP-specific packages:
google-cloud-aiplatform>=1.38.0
google-cloud-documentai>=2.20.0
google-cloud-storage>=2.14.0
google-cloud-firestore>=2.16.0

Step 5.3: Create a Dockerfile

A Dockerfile tells Cloud Build how to package your app into a container.

# Dockerfile
FROM python:3.11-slim

WORKDIR /app

# Install system dependencies
RUN apt-get update && apt-get install -y \
    build-essential \
    && rm -rf /var/lib/apt/lists/*

# Install Python dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Run the Streamlit app on Cloud Run's expected port
EXPOSE 8080
CMD ["streamlit", "run", "demo/streamlit_app.py", \
     "--server.port=8080", \
     "--server.address=0.0.0.0", \
     "--server.headless=true"]
What is Docker? Docker packages your entire app (code + dependencies + Python version) into a "container" - a portable box that runs exactly the same everywhere. Instead of telling Cloud Run "install Python, install these packages, set these env vars...", you give it one container that has everything pre-built. It's like shipping a fully assembled computer instead of shipping parts with assembly instructions.

Step 5.4: Create .dockerignore

# .dockerignore - don't send these to Cloud Build

__pycache__/
*.pyc
.env

Deploy to Cloud Run

Step 6.1: Build and deploy (one command!)

# This single command does everything:
# 1. Uploads your code to Cloud Build
# 2. Builds the Docker container in the cloud
# 3. Pushes it to Artifact Registry
# 4. Deploys it to Cloud Run
# 5. Gives you a public URL

gcloud run deploy pvr-chatbot \
  --source=. \
  --region=us-central1 \
  --service-account=pvr-chatbot-sa@$(gcloud config get-value project).iam.gserviceaccount.com \
  --allow-unauthenticated \
  --memory=2Gi \
  --cpu=2 \
  --timeout=300 \
  --set-env-vars="GCP_PROJECT_ID=$(gcloud config get-value project),GCP_LOCATION=us-central1,USE_VERTEX_AI=true,USE_GEMINI=true"
What is Cloud Run? It runs your Docker container as a web service. When someone visits the URL, Cloud Run starts your container, handles the request, and then shuts it down when idle (so you don't pay for idle time). The --allow-unauthenticated flag means anyone with the URL can access your Streamlit app (perfect for the interview demo). --memory=2Gi gives your app 2GB of RAM (needed for the embedding model).

Step 6.2: Get your live URL

# After deployment, Cloud Run gives you a URL like:
https://pvr-chatbot-xxxxx-uc.a.run.app

# Open it in your browser - your Streamlit app is live!
# You can share this URL with anyone.

# Check the status:
gcloud run services describe pvr-chatbot --region=us-central1
Interview power move: Open this URL on your phone during the interview. "Here's the same chatbot running on Google Cloud. You can try it on your phone right now." Interviewers LOVE live demos they can interact with.

What to say in the interview

When the interviewer asks about your architecture, walk them through it like this:

"A user uploads a PDF to Cloud Storage. That triggers Document AI to extract text and tables, even from scanned Telugu documents. The extracted text goes through my custom parser pipeline which classifies the document type and extracts structured metadata like property address, bedrooms, and price. The parsed chunks get embedded using Vertex AI's text-embedding-004 model and stored in Firestore with vector search indexing.

When a user asks a question, my hybrid search engine first extracts structured filters from the query using regex, then runs a vector similarity search on Firestore with those filters applied, then reranks results. The retrieved context gets injected into a query-specific prompt template, and Gemini Pro generates the answer. Post-processing verifies that any dollar amounts in the response actually exist in the source documents to prevent hallucination."

GCP Services Cheat Sheet for the Interview

If they ask about...You say...
"How do you handle scanned PDFs?""Document AI does OCR with ML-based table detection and supports 200+ languages including Telugu. It handles handwriting, stamps, and poor scan quality."
"How does it scale?""Cloud Run auto-scales from 0 to 1000 instances based on traffic. Firestore handles millions of vectors. Vertex AI embeddings and Gemini are fully managed and scale automatically."
"What about security?""Service accounts with least-privilege IAM roles. API keys in Secret Manager, not in code. Cloud Run supports VPC connectors for private networking. All data encrypted at rest and in transit."
"Cost?""Cloud Run scales to zero when idle, so no cost when not in use. Document AI free tier covers 1000 pages/month. Firestore free tier covers 1GB. For a typical real estate company with 500 documents, monthly cost would be under $50."
"Why not just use Langchain?""LangChain treats all PDFs as flat text and loses table structure. Our custom pipeline extracts structured metadata like bedrooms and price from specific document types, enabling hybrid search that combines metadata filtering with vector similarity. A query like '3-bedroom homes under $600K' actually filters on the number 3, not just hoping the embedding captures it."
"What about CI/CD?""Cloud Build triggers on git push to automatically rebuild and redeploy the Cloud Run service. I can set up a staging environment with a separate Cloud Run service pointing to the same Firestore."
