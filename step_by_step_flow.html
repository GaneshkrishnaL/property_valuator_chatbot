Ingestion Flow: appraisal_742_evergreen.pdf

We'll follow ONE PDF - the appraisal report for 742 Evergreen Terrace - through every step of the ingestion pipeline, showing you exactly what the data looks like at each stage.

Generate the PDF generate_sample_pdfs.py ▶
INPUT: Property data from Python dictionary
{
    "address": "742 Evergreen Terrace, Austin, TX 78701",
    "sqft": 2450,
    "bedrooms": 4,
    "bathrooms": 3,
    "lot_size": "0.28 acres",
    "year_built": 2018,
    "assessed_value": 525000,
    "listing_price": 549900,
    "neighborhood": "Downtown Austin"
}
WHAT HAPPENS: fpdf2 generates a multi-page PDF

The generate_appraisal() function creates a 3-page PDF with mixed content: narrative paragraphs about the property, a comparable sales TABLE with 4 rows of data, and a "Valuation Conclusion" section with the final estimated value. Headers ("CONFIDENTIAL") and footers ("Page 1 | Generated by...") are deliberately added to make it realistic.

OUTPUT: The actual PDF document
CONFIDENTIAL - Property Appraisal Report

Appraisal Report

742 Evergreen Terrace, Austin, TX 78701
Property Information

The subject property located at 742 Evergreen Terrace is a 4 bedroom, 3 bathroom single-family home built in 2018. Total living area: 2,450 square feet on a 0.28 acres lot in the Downtown Austin neighborhood.

Comparable Sales Analysis
AddressSale PriceSq Ft$/Sq Ft
750 Elm St$535,0002,380$225

Why this matters: Notice how this PDF has TEXT (paragraphs), TABLES (comparable sales), AND noise (headers/footers). Standard RAG tools would dump this all as one flat string, losing the table structure completely. That's the problem we're solving.
↓
Extract Text + Tables from PDF pdf_extractor.py ▶
INPUT: The raw PDF file (bytes on disk)

PyMuPDF (fitz) opens the PDF and processes each page separately.

WHAT HAPPENS - 3 things

1. Text extraction: page.get_text("text") pulls all text from each page.
2. Header/footer filtering: Lines containing "CONFIDENTIAL", "Page X |", or "Generated by" are removed. Without this, every chunk would contain "CONFIDENTIAL" and the search index would be polluted.
3. Table detection: page.find_tables() detects tables and extracts them as structured lists of rows, preserving the column layout.

OUTPUT: ExtractedDocument object
ExtractedDocument {
  filename: "appraisal_742_evergreen.pdf"
  page_count: 3
  pages: [
    ExtractedPage {
      text: "Appraisal Report\n742 Evergreen Terrace...\nThe subject property located at 742 Evergreen Terrace is a 4 bedroom, 3 bathroom single-family home built in 2018..."
      tables: [
        [["Address", "Sale Price", "Sq Ft", "$/Sq Ft"],  // header row
         ["750 Elm St", "$535,000", "2,380", "$225"],  // data row 1
         ["801 Oak Ave", "$612,000", "3,100", "$197"],
         ["723 Pine Rd", "$498,000", "2,200", "$226"]]
      ]
      has_tables: true
    }
  ]
  full_text: "Appraisal Report\n742 Evergreen Terrace..." (all pages joined)
  metadata: {filename: "appraisal_742_evergreen.pdf", pages: 3}
}
Key insight: Notice that the tables are extracted as STRUCTURED DATA (list of lists), NOT as flat text. The header row tells us what each column means. This is why the next step (parsing) can extract "$535,000" as a sale_price rather than just a random number.
↓
Classify Document Type document_classifier.py ▶
INPUT: The ExtractedDocument.full_text

The classifier scans the full text looking for keyword patterns specific to each document type.

WHAT HAPPENS - Pattern scoring
Scanning text for APPRAISAL signals:
  MATCH "appraisal report" → +0.35
  MATCH "appraiser" → +0.20
  MATCH "assessed value" → +0.15
  MATCH "valuation conclusion" → +0.20
  MATCH "subject property" → +0.10
  Structural bonus (3 pages, has tables) → +0.05
  TOTAL: 1.05 → capped at 1.0

Scanning text for LISTING signals:
  no match "property listing"
  no match "mls#"
  TOTAL: 0.10 (only "list price" partial match)

Winner: APPRAISAL with confidence 1.0
OUTPUT: ClassificationResult
ClassificationResult {
  document_type: DocumentType.APPRAISAL
  confidence: 1.0
  reasoning: "Classified as APPRAISAL based on 5 keyword signals: appraisal\\s+report, appraiser, assessed\\s+value. Score: 1.00"
}
Why this matters: Because we now know it's an APPRAISAL, we'll use the AppraisalParser in the next step. If we'd classified it as a LISTING, we'd use the ListingParser which looks for completely different fields (MLS#, days on market, etc.).
↓
Parse Structured Fields (AppraisalParser) custom_parsers.py ▶
INPUT: ExtractedDocument (text + tables from Step 2)

The AppraisalParser runs regex patterns to extract specific fields AND iterates through the extracted tables to pull out comparable sales data.

WHAT HAPPENS - Two parallel extractions

Regex extraction from text: Patterns like r"(\d+)\s*bedroom" find "4 bedroom" and extract the number 4. r"estimated\s+market\s+value.*?\$?([\d,]+)" finds "estimated market value of $525,000" and extracts 525000.

Table extraction: The parser looks at the extracted tables from Step 2. It finds the table whose headers contain "price" and "address" and iterates through each row, mapping columns to fields: Address column → comp address, Sale Price column → sale_price, etc.

Section splitting: Splits the text into sections (Introduction, Property Information, Valuation Conclusion) based on ALL-CAPS headers. Each section becomes a separate text unit for chunking.

OUTPUT: Parsed data with structured_fields + text_sections
{
  "structured_fields": {
    document_type: "APPRAISAL",
    property_address: "742 Evergreen Terrace, Austin, TX 78701",
    bedrooms: 4,
    bathrooms: 3,
    square_footage: 2450,
    year_built: 2018,
    assessed_value: 525000,
    neighborhood: "Downtown Austin",
    comparable_sales: [
      {address: "750 Elm St", sale_price: "$535,000", sqft: "2,380"},
      {address: "801 Oak Ave", sale_price: "$612,000", sqft: "3,100"},
      {address: "723 Pine Rd", sale_price: "$498,000", sqft: "2,200"}
    ]
  },
  "text_sections": [
    {title: "Introduction", text: "Appraisal Report\n742 Evergreen Terrace..."},
    {title: "Property Information", text: "The subject property located at..."},
    {title: "Valuation Conclusion", text: "Based on the comparable sales analysis..."}
  ]
}
This is the magic step. We've turned an unstructured PDF into structured data. The number "4" isn't just text anymore. It's bedrooms: 4 as metadata. Later, when someone asks "3-bedroom homes under $600K", we can filter on bedrooms != 3 and EXCLUDE this property instantly, without even checking the text.
↓
Chunk into Searchable Pieces chunker.py ▶
INPUT: Parsed data from Step 4 (structured_fields + text_sections)

Each text_section becomes one or more chunks. The structured_fields become metadata attached to EVERY chunk from this document.

WHAT HAPPENS

Section → Chunk mapping: "Introduction" section (short) → 1 chunk. "Property Information" section (short) → 1 chunk. "Valuation Conclusion" section (short) → 1 chunk.

Structured summary chunk: An extra chunk is created from the structured fields as readable text: "Property Summary:\nProperty Address: 742 Evergreen Terrace...\nBedrooms: 4\nAssessed Value: $525,000..."

Metadata attachment: Every chunk gets the same metadata: bedrooms=4, price=525000, neighborhood="Downtown Austin", etc.

OUTPUT: 4 DocumentChunks (3 sections + 1 summary)
Chunk 1 (section: Introduction)
text: "[Introduction]\nAppraisal Report\n742 Evergreen Terrace, Austin, TX..."
metadata: {bedrooms: 4, price: 525000, neighborhood: "Downtown Austin", document_type: "APPRAISAL", property_address: "742 Evergreen Terrace..."}

Chunk 2 (section: Property Information)
text: "[Property Information]\nThe subject property located at 742 Evergreen Terrace is a 4 bedroom, 3 bathroom..."
metadata: {bedrooms: 4, price: 525000, neighborhood: "Downtown Austin", ...same as above}

Chunk 3 (section: Valuation Conclusion)
text: "[Valuation Conclusion]\nBased on the comparable sales analysis and property condition assessment, the estimated market value of 742 Evergreen Terrace is $525,000..."
metadata: {same}

Chunk 4 (section: Property Summary - auto-generated)
text: "Property Summary:\nProperty Address: 742 Evergreen Terrace, Austin, TX 78701\nAssessed Value: $525,000\nSquare Footage: 2,450\nBedrooms: 4\nBathrooms: 3\nYear Built: 2018\nNeighborhood: Downtown Austin"
metadata: {same}
Two levels of search: The text is what gets embedded and matched semantically (meaning). The metadata is what gets filtered exactly (numbers). When you search, metadata filters first, THEN vector similarity runs on the remaining chunks.
↓
Embed Text into Vectors embeddings.py ▶
INPUT: The .text field of each DocumentChunk

Each chunk's text gets converted into a 384-dimensional vector (array of 384 numbers) by the sentence-transformers model.

WHAT HAPPENS

The model all-MiniLM-L6-v2 reads the text and produces a vector that captures the MEANING. Similar texts produce similar vectors. "Property valued at $525K" and "What is this home worth?" would produce vectors that are close together in 384-dimensional space, even though the words are different.

OUTPUT: Vector for each chunk
Chunk 1 text: "[Introduction]\nAppraisal Report\n742 Evergreen..."
Chunk 1 vector: [0.023, -0.118, 0.445, 0.087, -0.234, ..., 0.067] (384 numbers)

Chunk 3 text: "[Valuation Conclusion]\n...estimated market value...is $525,000..."
Chunk 3 vector: [0.156, -0.042, 0.312, 0.198, -0.089, ..., 0.145] (384 numbers)
Why vectors? You can't mathematically compare "What's this house worth?" with "estimated market value of $525,000" using keyword matching. But as vectors, you compute the cosine similarity (dot product) and get a number: 0.82 = very similar. That's how semantic search works.
↓
Store in ChromaDB vector_store.py ▶
INPUT: Vectors + text + metadata for all 4 chunks

ChromaDB stores each chunk with three things: the vector (for similarity search), the raw text (to return as context), and the metadata (for filtering).

WHAT HAPPENS

ChromaDB's add() method stores everything on disk. When you later call search(), it can first filter by metadata (e.g., bedrooms=4) and then find the closest vectors in the filtered subset. This is much faster than searching all chunks.

OUTPUT: Chunks stored on disk, ready to search
ChromaDB collection: "property_docs"
  Total chunks stored: ~45 (4 from this PDF + ~41 from other 10 PDFs)
  Each chunk has: vector (384-dim) + text + metadata (bedrooms, price, neighborhood...)

Now the ingestion is DONE. All 11 PDFs have been processed and stored.
Time taken: ~6-10 seconds total.
Query Flow: "What is the estimated value of 742 Evergreen Terrace?"

We'll follow this one query through the entire Retrieval → Augmentation → Generation pipeline, showing you exactly what happens at each step.

Query Enters the Chatbot chatbot.py → ask() ▶
INPUT: Raw user query string
"What is the estimated value of 742 Evergreen Terrace?"

The ask() method receives this raw query. It does NOT add conversation history here. The raw query goes directly to the search system to avoid filter contamination (the bug we fixed).

↓
RETRIEVAL: Extract Filters from Query search.py → extract_query_filters() ▶
INPUT: The raw query string

Regex patterns scan the query looking for structured information: bedroom counts, price ranges, neighborhoods, and addresses.

WHAT HAPPENS - Regex pattern matching
Checking for bedrooms: r"(\d+)\s*bed" → no match
Checking for price cap: r"under|below.*\$([\d,]+)" → no match
Checking for neighborhood: "downtown austin", "south congress"... → no match
Checking for address: r"(\d+\s+[\w\s]+terrace)" → MATCH: "742 evergreen terrace"
Checking for doc type: "value" found → MATCH: document_type = "APPRAISAL"
OUTPUT: QueryFilters
QueryFilters {
  bedrooms: None,
  max_price: None,
  neighborhood: None,
  property_address: "742 evergreen terrace",
  document_type: "APPRAISAL"
}
Notice: The filter extraction found two things: a property address and a document type hint. It did NOT find bedrooms, price, or neighborhood filters. This means the metadata filter will narrow results to APPRAISAL documents only, and the reranker will boost results matching "742 evergreen terrace" in their address.
↓
RETRIEVAL: Build ChromaDB Filter search.py → build_chroma_where() ▶
INPUT: QueryFilters from Step 2
OUTPUT: ChromaDB where clause
Only document_type filter is applied (address matching happens in reranking):

where = {"document_type": "APPRAISAL"}

This tells ChromaDB: "only search within APPRAISAL chunks, ignore listings, market analysis, etc."
This narrows ~45 total chunks down to ~12 appraisal chunks.
↓
RETRIEVAL: Embed Query + Vector Search embeddings.py + vector_store.py ▶
STEP A: Embed the query
Query: "What is the estimated value of 742 Evergreen Terrace?"
Query vector: [0.189, -0.067, 0.398, 0.112, ..., 0.203] (384 numbers)
STEP B: Search ChromaDB (filtered to APPRAISAL chunks only)

ChromaDB computes the cosine similarity between the query vector and every APPRAISAL chunk's vector. Chunks about "valuation" and "742 Evergreen" score highest because their meanings are closest to the query.

OUTPUT: Top 10 raw results (before reranking)
#1 score=0.82 | Chunk: Valuation Conclusion | 742 Evergreen | "...estimated market value of $525,000..."
#2 score=0.78 | Chunk: Property Summary | 742 Evergreen | "Property Address: 742 Evergreen...Assessed Value: $525,000..."
#3 score=0.71 | Chunk: Property Information | 742 Evergreen | "The subject property located at 742..."
#4 score=0.65 | Chunk: Valuation Conclusion | 1510 Maple Dr | "...estimated market value of $415,000..."
#5 score=0.62 | Chunk: Property Summary | 1510 Maple Dr | "Property Address: 1510 Maple..."
...and 5 more lower-scoring results
Notice: The top 3 results are all about 742 Evergreen because the query mentioned that address AND the text content is about valuation. But results #4-5 are about a DIFFERENT property (1510 Maple) because their Valuation Conclusion text is semantically similar to the query. This is where reranking helps.
↓
RETRIEVAL: Rerank Results search.py → _rerank() ▶

Result #4 (1510 Maple - Valuation Conclusion):
  Base score: 0.65
  +0.00 address match: "742 evergreen" NOT in "1510 Maple"
  +0.05 document type match: APPRAISAL = APPRAISAL
  Final: 0.70
OUTPUT: Reranked results (742 Evergreen chunks all at the top)

(+ more below threshold)
Result of retrieval layer: We have 5-10 relevant chunks, with the 742 Evergreen ones at the top. The 1510 Maple chunks are included as additional context (they're still above the 0.3 threshold). These results now get passed to the Augmentation layer.
↓
RETRIEVAL → AUGMENTATION LAYER
AUGMENTATION: Classify Query + Pick Template prompt_templates.py → classify_query() ▶
INPUT: The original query string
WHAT HAPPENS - Keyword matching
Query: "What is the estimated value of 742 Evergreen Terrace?"

Checking VALUATION keywords: "value", "worth", "price of", "appraisal"...
  MATCH: "value" → QueryType.VALUATION

Selected template: VALUATION_TEMPLATE
This template includes the critical instruction: "Look for explicitly stated values first. Do NOT compute your own estimate."
Why separate templates? If this were a market trends question, we'd pick MARKET_ANALYSIS_TEMPLATE which tells the LLM to focus on statistics and trends. Different questions need different instructions. Without this, the LLM might give a market overview when you asked for a specific property value.
↓
AUGMENTATION: Build the Full LLM Prompt prompt_templates.py → build_prompt() ▶
WHAT HAPPENS - Assembling 3 pieces

The final prompt sent to GPT-4o-mini is assembled from three pieces: the system prompt (who you are + rules), the retrieved chunks formatted as context, and the user's question.

OUTPUT: System Prompt
SYSTEM PROMPT
"You are a property valuation assistant for the Austin, Texas real estate market...

CRITICAL RULES:
1. ONLY use information from the provided context documents. Never use your general knowledge.
2. ALWAYS cite which document each piece of information comes from [Source: filename].
3. If context doesn't have enough info, say 'I don't have sufficient data...'
4. Never invent or estimate property values unless data explicitly supports it.
5. When discussing prices, specify whether it's assessed value, listing price, sale price..."
OUTPUT: User Prompt (template + context + question)
USER PROMPT
"Based on the following property documents, answer the user's valuation question.

IMPORTANT: Look for explicitly stated values first...

RETRIEVED CONTEXT:
--- Document 1 [Source: appraisal_742_evergreen.pdf] (Type: APPRAISAL, Relevance: 1.00) ---
[Valuation Conclusion] Based on the comparable sales analysis and property condition assessment, the estimated market value of 742 Evergreen Terrace is $525,000...

--- Document 2 [Source: appraisal_742_evergreen.pdf] (Type: APPRAISAL, Relevance: 0.98) ---
Property Summary: Property Address: 742 Evergreen Terrace...Assessed Value: $525,000...Bedrooms: 4...

--- Document 3 [Source: appraisal_742_evergreen.pdf] (Type: APPRAISAL, Relevance: 0.91) ---
[Property Information] The subject property located at 742 Evergreen Terrace is a 4 bedroom...

... (more documents) ...

USER QUESTION: What is the estimated value of 742 Evergreen Terrace?"
This is "Augmentation" in RAG: We're "augmenting" the LLM's prompt with REAL RETRIEVED DATA. The LLM doesn't need to know anything about Austin real estate from its training. It just reads the documents we gave it and answers from those. That's why it can't hallucinate a value, the prompt explicitly says "look for stated values first."
↓

The complete prompt is sent to OpenAI's GPT-4o-mini API (temperature=0.3 for factual accuracy). The model reads the retrieved documents in the context and generates a natural language answer.
